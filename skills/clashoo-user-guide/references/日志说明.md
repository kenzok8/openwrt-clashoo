# 日志说明

Clashoo 的日志分为几类。不同日志对应不同问题，排查时不要混在一起看。

日志在 `系统 -> 日志` 页面查看，物理位置分别是：

| 日志 | 路径 | 内容 |
|---|---|---|
| 插件日志 | `/usr/share/clashoo/clashoo.txt` | Clashoo 自身的启停、配置准备等步骤（界面主日志） |
| 核心日志 | `/var/log/clashoo/core.log` | mihomo / Smart / sing-box 内核原生输出与报错 |
| 更新日志 | 内核 / 面板 / 资源各自更新脚本的输出 | 下载与更新任务的结果 |

> 核心日志里 mihomo 的时间是 UTC，Clashoo 在界面里会自动 +8 转成北京时间显示。

## 插件日志

插件日志记录 Clashoo 自身行为，例如：

- 启动任务
- 停止任务
- 配置准备
- RPC 操作

如果按钮无响应、启动失败、停止失败，优先看插件日志。

## 核心日志

核心日志来自 mihomo、Smart 或 sing-box。

常见内容：

- 节点连接
- DNS 查询
- 规则匹配
- 入站 / 出站连接
- 核心错误

核心日志比较详细，属于正常现象。排查代理失败时重点看 `ERROR`、`FATAL`、`timeout`、`connection reset` 等关键字。

## 更新日志

更新日志记录资源和组件更新任务，例如：

- 内核下载
- 面板更新
- GeoIP 更新
- 大陆白名单更新

如果点击更新后没有变化，看更新日志。

## 常见日志含义

`health check failed`

健康检查失败，可能是 DNS、显式代理或透明代理任一环节异常。

`routing_mark is conflict with tun.auto_redirect`

sing-box 配置中 TUN auto_redirect 和 routing_mark 冲突，需要使用修复后的 Clashoo 版本，并重新生成运行配置。

`context deadline exceeded`

连接或 DNS 请求超时，可能是节点、DNS 或网络不可达。

`connection reset by peer`

连接被对端重置，可能是节点不可用、协议不兼容或服务端拒绝。

## 排查建议

- 启动失败：先看插件日志，再看核心日志
- 国外不通：看核心日志中的节点连接错误
- 国内不通：看 DNS 和直连规则
- 更新失败：看更新日志
